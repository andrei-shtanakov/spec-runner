| | |
|---|---|

| Status | Approved (brainstorming complete; awaiting user review of spec) |

## Context

A monorepo test run of `spec-runner run --all` against the appgraph project (2026-05-29) exposed several gaps in spec-runner 2.2.2's user feedback and recovery semantics:

- 4 of 14 tasks failed and `status` only said `Last error: Unknown error...`. The real cause (codex OpenAI quota: "try again at 9:54 AM") was buried in `spec/.executor-logs/TASK-*.log` under `=== STDERR ===`.
- After the run stopped at `consecutive_failures 12/2`, a subsequent `run --all` could not resume cleanly — failed-status tasks stayed failed and the counter persisted. The user had to `retry` individual tasks.
- Per-task progress mirroring landed in 2.2.2 (`7aceddc feat(obs): mirror compact progress to stderr in non-TUI runs`), but stops at task-level events; there is no sub-stage visibility (sync deps / codex / tests / lint / review).
- Two unrelated bugs were also discovered: codex CLI auto-detect builds `codex -p <prompt>` but codex's `-p` is `--profile`, not the prompt; and git-automation defaults (`create_git_branch=true`, `auto_commit=true`) committed 171 files across an entire monorepo when the project lives in a subdir of a larger git repo.

This spec captures the 5 user-requested UX/behavior features plus the 2 bonus fixes for one minor release.

## Scope

**In-scope:**
- F1: Show spec-runner version in `status` output.
- F2: Show short, human-readable error reason in `status` (with provider patterns + stderr-tail fallback).
- F3: `run --all` resets `failed → pending` and `consecutive_failures → 0` by default; opt-out via `--no-reset-failed`.
- F4: Mirror per-task **sub-stages** (sync_deps, codex, parse, tests, lint, commit, merge, review) to stderr.
- F5: When a task fails a "second pass" (failed before, failed again this run), emit a hint pointing at its log file — both during the run and in `status`.
- B1: Fix codex adapter in `runner.py::build_cli_command` to use `codex exec [-m MODEL] <PROMPT>`.
- B2: Auto-default git automation OFF for subdir-projects (project_root ≠ git repo toplevel), unless the user explicitly opts in.

**Out-of-scope:**
- TUI redesign, new provider adapters (we only fix codex, not add new ones).
- Migration framework or schema versioning beyond idempotent ALTER.
- Re-architecting state.py / execution.py.
- Plugin-based error pattern registries (the pattern list is in-source, extend by editing the list).

## Decisions log

| # | Decision | Picked | Why |
|---|---|---|---|
| D1 | Release packaging | One release `2.3.0` with all 5 features + 2 bonus fixes | All discovered in same testing pass; single changelog; lets us drop the workaround config in appgraph. |
| D2 | Error reason strategy | Hybrid: regex pattern library + last-5-lines-of-stderr fallback | Provider patterns are short; fallback covers unknown CLIs. |
| D3 | `run --all` reset semantics | Default-reset + `--no-reset-failed` opt-out flag | Common case is "user re-runs after fixing something or after quota reset"; opt-out preserved for sticky-failed workflows. |
| D4 | Stage logging granularity | Mirror **all** sub-stages to stderr | User explicitly chose visibility over compactness. |
| D5 | Second-pass hint surface | Both during-run warning AND persistent in `status` | Not missed in-flight; visible afterward without re-running. |

## Design — User-visible (UX)

### F1. Version line in `status`

First line of `spec-runner status` becomes the version:

```
📊 spec-runner v2.3.0
==================================================
Tasks in spec:         14
...
```

Implementation: `cli_info.py` prints `from spec_runner import __version__` in the existing status header function.

### F2. Error surfacing in `status`

Two related additions, both driven by the same pattern-library + fallback (see Internals → errors.py):

**F2a. Per-task error display.** Replace `Last error: Unknown error...` with `[error_kind] human message` and an optional `at: stage` tag:

```
❌ TASK-009: failed (3 attempts) [at: codex]
   Last error: [rate_limit] OpenAI usage limit — try again at 9:54 AM
```

Source: `attempts.error_kind` (categorical) + `attempts.error` (human text from pattern template OR stderr tail). Stage from `attempts.error_stage`.

Fallback for legacy rows (NULL columns): show old format unchanged.

**F2b. Stop-reason summary in `status` header.** When the most recent run ended abnormally, `cli_info.py` prints a single warning line under the title:

```
📊 spec-runner v2.3.0
⚠️ Last run stopped: max_consecutive_failures reached (12/2)
==================================================
```

or

```
⚠️ Last run stopped: rate_limit (codex) — try again at 9:54 AM
```

If `last_run_stop_reason == "completed"` (or absent), no line is printed.

Source: `executor_meta.last_run_stop_reason` + `executor_meta.last_run_stop_detail`.

### F3. `run --all` reset (UX consequence)

No standalone UX surface — F3 is a behavior change (see Internals). UX side-effect:
- The ⚠️-stop-reason line (F2b) gets refreshed by every run-end, so after a fresh `run --all`, it reflects the new outcome.
- The 💡-second-pass hint (F5) appears for any task that fails *again* across runs.
- New CLI flag `--no-reset-failed` shows up in `spec-runner run --help`.

### F4. Per-stage progress mirror (extends 2.2.2 `feat(obs)`)

Extend the existing `feat(obs): mirror compact progress to stderr` (2.2.2) with one short line per sub-stage transition:

```
[05:30:29] [TASK-001] 🚀 Starting: Tooling, dependencies & package bootstrap
[05:30:29] [TASK-001] ⏳ stage: sync_deps
[05:30:31] [TASK-001] ⏳ stage: codex
[05:33:42] [TASK-001] ⏳ stage: parse
[05:33:42] [TASK-001] ⏳ stage: tests
[05:33:55] [TASK-001] ⏳ stage: lint
[05:33:58] [TASK-001] ⏳ stage: review
[05:36:56] [TASK-001] ✅ Completed in 209.2s
```

Stages emitted only when they actually run:
- `branch` and `merge` are emitted only when `create_git_branch=true` (the merge block in `hooks.py` is gated behind `create_git_branch`).
- `commit` is emitted only when `auto_commit=true`.
- `tests` / `lint` skipped when `run_tests_on_done` / lint hook are off.

### F5. "Analyze logs" hint for second-pass failures

A task is "second-pass failed" when:
- it had `status == "failed"` immediately before the current `run --all` (snapshot taken before the F3 reset), AND
- it ended this run with `status == "failed"` (retry budget exhausted again).

Two surfaces:
- During the run, immediately after the second-pass failure is recorded:
  ```
  💡 [TASK-009] repeated failure — review logs at spec/.executor-logs/TASK-009-*.log
  ```
- In `status` Task History, a 💡-block under that task:
  ```
  ❌ TASK-009: failed (3 attempts) [at: codex]
     Last error: [rate_limit] OpenAI usage limit — try again at 9:54 AM
     💡 Repeated failure across runs — review:
        spec/.executor-logs/TASK-009-*.log
  ```

The second-pass set is cleared at the start of the next `run --all` (along with the F3 reset) so it does not accumulate stale entries.

## Design — Internals

### State schema additions

Idempotent on-startup migration in `state.py`:

```python
# attempts: add error_kind and error_stage if missing
try: conn.execute("ALTER TABLE attempts ADD COLUMN error_kind TEXT")
except sqlite3.OperationalError: pass
try: conn.execute("ALTER TABLE attempts ADD COLUMN error_stage TEXT")
except sqlite3.OperationalError: pass
```

`executor_meta` (key/value) gets three new keys (no schema change, just new keys):
- `last_run_stop_reason` ∈ {`completed`, `max_consecutive_failures`, `error_rate_limit`, `error_auth`, `error_network`, `error_cli_error`, `error_unknown`, `killed`}
- `last_run_stop_detail` (free text, e.g., "try again at 9:54 AM" or "12/2")
- `second_pass_fail_tasks` (CSV of task IDs)

Legacy rows / missing keys → NULL → UI falls back to current behavior.

### Error pattern library — `src/spec_runner/errors.py` (new)

```python
from dataclasses import dataclass
import re

@dataclass(frozen=True)
class ErrorPattern:
    kind: str
    regex: re.Pattern
    template: str   # supports {0}, {1} from regex groups

PATTERNS: list[ErrorPattern] = [
    # codex / OpenAI quota
    ErrorPattern("rate_limit",
        re.compile(r"hit your usage limit.*?try again at ([\d:]+\s*[AP]M)", re.S),
        "OpenAI usage limit — try again at {0}"),
    # generic rate-limit
    ErrorPattern("rate_limit",
        re.compile(r"rate[_\s-]?limit", re.I),
        "Rate limit hit"),
    # auth
    ErrorPattern("auth",
        re.compile(r"unauthor|invalid api key|forbidden", re.I),
        "Authentication failed"),
    # network
    ErrorPattern("network",
        re.compile(r"ECONNREFUSED|timed out|name or service not known|dns", re.I),
        "Network error"),
    # generic CLI error line
    ErrorPattern("cli_error",
        re.compile(r"^error:\s*(.+)$", re.M),
        "{0}"),
]

STDERR_TAIL_LINES = 5

def classify(stderr: str, returncode: int) -> tuple[str, str]:
    """Return (kind, human_message). Falls back to ('unknown', last N stderr lines)."""
    for p in PATTERNS:
        m = p.regex.search(stderr)
        if m:
            try:
                return p.kind, p.template.format(*m.groups())
            except IndexError:
                return p.kind, p.template
    tail = "\n".join(stderr.strip().splitlines()[-STDERR_TAIL_LINES:])
    return "unknown", tail or f"CLI exited with code {returncode}"
```

Used in `execution.py` at the existing "Unknown error" site, and once at the end of `_run_tasks` to decide `last_run_stop_reason/detail`.

Extending: add a new `ErrorPattern(...)` to the list. No registry / DI.

### Stage tracking — `src/spec_runner/stages.py` (new) + explicit passing

```python
STAGES = ("sync_deps", "branch", "codex", "parse",
          "tests", "lint", "commit", "merge", "review")

class StageReporter:
    """One per task. Threaded explicitly through execution; safe with max_concurrent>1."""
    def __init__(self, task_id: str, mirror_fn):
        self.task_id = task_id
        self.mirror = mirror_fn
        self.current: str | None = None

    def enter(self, name: str) -> None:
        assert name in STAGES, f"unknown stage: {name}"
        self.current = name
        self.mirror(f"[{self.task_id}] ⏳ stage: {name}")
```

Lifecycle:
- Created in `execute_task` for each task with `mirror_fn = log_progress` (the same callback the 2.2.2 obs commit uses).
- Passed to `pre_start_hook`, `execute_task`'s subprocess section, `post_done_hook`, `run_review`.
- On exception/non-zero exit, `attempts.error_stage = reporter.current`.

No thread-locals (concurrent runs would interfere); explicit parameter passing.

### `run --all` reset — `cli.py::_run_tasks`

```python
reset_enabled = (
    getattr(args, "all", False)
    and not getattr(args, "no_reset_failed", False)
)
previously_failed: set[str] = set()
if reset_enabled:
    previously_failed = state.reset_failed_to_pending()
    state.consecutive_failures = 0
    state.clear_second_pass_fails()  # see F5 cleanup
    state._save()
```

New methods on `ExecutorState`:
- `reset_failed_to_pending() -> set[str]` — updates rows `status='failed' → 'pending'`, returns the set of task IDs that were reset (used by F5).
- `clear_second_pass_fails()` — removes the `second_pass_fail_tasks` meta key.

CLI: add `--no-reset-failed` to the `run` subparser in `cli.py`. Single-task runs (`run TASK-X`) and `retry` are unaffected.

### Second-pass detection

The `previously_failed` set (in-memory) is captured BEFORE the reset (in `reset_failed_to_pending`). After each task finishes:

```python
if task.id in previously_failed and final_status == "failed":
    log_progress(
        f"💡 [{task.id}] repeated failure — review logs at "
        f"{config.logs_dir}/{task.id}-*.log"
    )
    state.add_second_pass_fail(task.id)
```

`add_second_pass_fail` appends to `executor_meta.second_pass_fail_tasks` (CSV).

`cli_info.py` reads the CSV and, in Task History, prints a 💡-block under each matching task.

### Stop reason capture — `cli.py::_run_tasks` finally-block

```python
try:
    # ... iterate tasks ...
finally:
    if state.consecutive_failures >= config.max_consecutive_failures:
        reason = "max_consecutive_failures"
        detail = f"{state.consecutive_failures}/{config.max_consecutive_failures}"
    elif last_failure_kind:  # from the last failed attempt this run
        reason = f"error_{last_failure_kind}"
        detail = last_failure_message
    else:
        reason = "completed"
        detail = ""
    state.set_meta("last_run_stop_reason", reason)
    state.set_meta("last_run_stop_detail", detail)
```

`set_meta`/`get_meta` are thin wrappers over the existing meta table operations.

### B2. Subdir-project git automation auto-default — `src/spec_runner/config.py`

```python
def _detect_subdir_repo(project_root: Path) -> Path | None:
    """Return git repo toplevel if project_root is a subdir of a git repo, else None."""
    try:
        out = subprocess.run(
            ["git", "-C", str(project_root), "rev-parse", "--show-toplevel"],
            capture_output=True, text=True, check=True,
        ).stdout.strip()
        toplevel = Path(out).resolve()
        return toplevel if toplevel != project_root.resolve() else None
    except (subprocess.CalledProcessError, FileNotFoundError):
        return None

def _user_set(yaml_config: dict, args: argparse.Namespace, key: str) -> bool:
    """True if user explicitly set this in YAML or CLI (not just default)."""
    if yaml_config.get(key) is not None:
        return True
    return getattr(args, key, None) not in (None, False)
```

Applied in `build_config()` after the YAML+CLI merge:

```python
git_root = _detect_subdir_repo(config.project_root)
if git_root is not None:
    flipped = []
    if not _user_set(yaml_config, args, "create_git_branch"):
        config.create_git_branch = False
        flipped.append("create_git_branch")
    if not _user_set(yaml_config, args, "auto_commit"):
        config.auto_commit = False
        flipped.append("auto_commit")
    logger.warning(
        "subdir_project_detected",
        project_root=str(config.project_root),
        git_root=str(git_root),
        defaulted_off=flipped,
        override_hint="set create_git_branch/auto_commit=true in YAML to opt-in",
    )
```

Explicit user `true` is respected (with the warning still emitted so they are aware).

## Risks & rollback

- **Schema migration:** additive only (two `ALTER TABLE ADD COLUMN` + new meta keys); idempotent. Old `attempts` rows with NULL `error_kind`/`error_stage` keep showing in the legacy format.
- **Default behavior change for `run --all`:** the reset is a semantic change. Mitigated by the `--no-reset-failed` opt-out and a clear CHANGELOG entry.
- **Subdir auto-default (B2):** could surprise users who actually wanted repo-wide git ops. Mitigated by:
  - Only flips when the user did not explicitly set the flag.
  - Warning log every run when triggered.
  - CHANGELOG entry calling it out.
- **codex adapter (B1):** existing `command_template` overrides keep working unchanged.
- **Rollback:** `uv tool install spec-runner==2.2.2 --reinstall` reverts. State DB is forward-compatible (extra columns are simply ignored by 2.2.2).
